# NOHTUS WMS Master Memory

이 문서는 NOHTUS WMS 프로젝트의 기준 문서이다. 새 채팅, 새 개발자, 새 AI가 작업을 이어받을 때 가장 먼저 읽는다.

## 프로젝트 개요

- 프로젝트명: NOHTUS WMS
- 목적: 노투스팜 물류팀의 입고, 출고, 이동, 재고, 실사, 제품매칭, 마감 업무를 보조하는 내부 WMS
- 현재 기술: Python, Streamlit, SQLite
- 현재 안정 기준: v4.9 RC3.x
- 운영 방식: ERP를 대체하지 않고, 현장 재고 판단과 작업 속도를 보조한다.

## Core Freeze

아래 기능은 안정화된 핵심 기능이다. 리팩토링 시 동작을 바꾸지 않는다.

- 입고 등록의 도면 클릭 선택
- JS Bridge를 통한 Streamlit query parameter 연동
- 입고 위치 자동 선택
- 로케이션 맵 상세 보기
- 로케이션 맵 제품명 클릭 후 검색 자동 실행
- ERP명/비자료명 기반 제품 매칭
- 재고 이동 및 사업장 이동
- 출고지시 저장 및 마감 체크

핵심 함수 및 파일:

- `app.py`: 현재 메인 애플리케이션. 기능 이동 전까지 안정 기준으로 유지한다.
- `styles.py`: 전역 Streamlit CSS. 사이드바 CSS가 본문 버튼에 침범하지 않게 한다.
- `inbound_map.py`: 입고 등록 도면과 JS Bridge. 특히 클릭 연동은 신중히 다룬다.
- `render_inbound_quick_location_map()`
- `location_picker()`
- `_apply_inbound_location_pending()`
- `_inbound_js_loc_changed()`

## UI 원칙

- 업무용 UI이므로 빠르고 단순해야 한다.
- 사이드바는 남색 테마를 유지한다.
- 사이드바 메뉴는 좌측 정렬한다.
- 로케이션 맵은 오른쪽이 잘리지 않아야 한다.
- UI 수정은 기능 로직과 분리해서 처리한다.
- 버튼, 카드, 표는 현장 작업자가 빠르게 스캔할 수 있어야 한다.

## 사업장 규칙

사업장은 다음 4개를 기준으로 한다.

- 노투스팜
- 노투스
- NOH
- 비자료

입고 등록에서는 `등록대기`도 사용할 수 있다.

## 로케이션 규칙

대표 구역:

- A1, A2, B1, B2, C1, C2, D1, E1
- F1: 비자료 계열
- G1, G2
- REC: 매입등록대기
- Q: 유통기한 임박
- P: 수출대기
- R1, R2
- X1, X2
- N: 기타 위치

기타 위치:

- 홍보물랙
- 회색 카트
- 오른쪽 창고
- 사무실(4층)

## DB 원칙

- `inventory`: 현재 재고
- `transactions`: 모든 재고 변화 이력
- `products`: 표준제품명 및 ERP명/비자료명 매칭
- `customers`: 거래처 정보
- 삭제보다 이력 보존을 우선한다.
- 제품명 변경 시 이미 등록된 재고와 이력 화면에도 반영되어야 한다.

## Git 운영

- `main`: 안정 버전
- `refactor/*`: 구조 개선
- `feature/*`: 신규 기능
- `fix/*`: 버그 수정

리팩토링은 작은 단위로 나누고, 각 단계에서 문법 검사 및 핵심 화면 수동 확인을 한다.

## 리팩토링 방향

1. 설정/상수 분리: `nohtus/config.py`
2. DB helper 분리: `nohtus/db.py`
3. 메뉴 구조 분리: `nohtus/navigation.py`
4. Excel import/export 함수 분리
5. 제품 매칭 서비스 분리
6. 입고/출고/이동/실사/마감 화면을 `pages/` 단위로 분리
7. 마지막에 `app.py`는 page routing 중심으로 축소

## 작업 규칙

- 기존 기능을 먼저 이해한 뒤 수정한다.
- UI 변경과 DB/업무 로직 변경을 한 커밋에 섞지 않는다.
- `app.py`를 크게 줄이더라도 한 번에 모든 함수를 옮기지 않는다.
- 리팩토링 후에는 최소한 Python 문법 검사와 주요 화면 수동 확인을 한다.
- 사용자가 현장에서 바로 쓰는 흐름을 최우선으로 한다.
